# 思考过程标准化

> 从本次创世文档讨论中提炼的思考流程。适用于任何产品设计和架构决策，不限于语言学习项目。

---

## 思考流程总览

```
Phase 1: 定义问题
  ① 从真实痛点出发
  ② 定义哲学锚点

Phase 2: 构建模型
  ③ 拆解领域的基本要素
  ④ 持续做减法
  ⑤ 区分核心与应用层

Phase 3: 验证与对抗
  ⑥ 用理论验证直觉
  ⑦ 接受外部挑战，坚守关键判断

Phase 4: 收敛
  ⑧ 追求优雅：最少概念，最广覆盖
```

---

## Phase 1: 定义问题

### ① 从真实痛点出发

**原则：** 不要问"市面上有什么，我要做哪些"。要问"我遇到了什么真实问题，现有工具为什么解决不了"。

**怎么做：**
- 描述你自己的真实使用场景和遇到的阻碍
- 每个功能必须能追溯到一个真实痛点，不存在"可能有用"的功能
- 如果你不是目标用户，去找到真正的目标用户，听他们的痛点

**本次案例：**
起点不是"做一个语言学习 App"，而是"我学了两门语言，知识散落在 AI 对话中，无法复习、搜索、量化"。

---

### ② 定义哲学锚点

**原则：** 在讨论任何功能之前，先回答"为什么"。哲学确定后，功能设计自然有取舍标准。

**怎么做：**
- 问自己：这个产品存在的根本理由是什么？
- 用一句话表达核心信念
- 后续所有决策都用这句话作为裁判标准

**本次案例：**
"AI 说的并不是你内化的。这个系统要解决的，就是从 AI 的信息到人的内化之间的鸿沟。"——后续所有设计分歧都回到这句话判断。

---

## Phase 2: 构建模型

### ③ 拆解领域的基本要素

**原则：** 不要从功能出发设计数据结构。先问"这个领域的本质是什么，由哪些基本要素构成"。

**怎么做：**
- 列出领域中所有你能想到的要素
- 问每一个要素：它是独立存在的实体，还是某个实体的属性？
- 问每一个要素：它属于领域本身，还是属于使用领域的方式？

**本次案例：**
语言知识的要素：单词、语法、读音、语体、语感、例句、使用条件... → 追问后发现：单词和语法是独立实体（原子），其他都是实体的特征（属性）或实体之间的联系（关系）。

---

### ④ 持续做减法

**原则：** 每引入一个概念，都要问"这个是必须的吗"。如果去掉它系统依然完整，就不应该存在。

**怎么做：**
- 对每个概念执行"删除测试"：删掉它，核心还完整吗？
- 如果一个概念的作用可以被另一个概念覆盖，合并它
- 如果一个概念既能放在核心也能放在应用层，放应用层

**本次案例：**
```
六个概念 → 问"例句是必须的吗"
  → 独立概念变成"分析记录"，归入应用层
五个概念 → 问"标签是知识本身吗"
  → 不是，是组织方式，归入应用层
四个概念 → 问"轨迹是知识本身吗"
  → 不是，是学习行为，归入应用层
三个概念 → 原子、属性、关系，删掉任何一个都不完整
```

---

### ⑤ 区分核心与应用层

**原则：** 不要把所有东西平列。先找到不变的核心，再把其他东西归为可扩展的应用层。

**判断标准：**
- **核心**：不变的、本质的、少一个就不完整的、跨场景通用的
- **应用层**：为功能服务的、可以按需增减的、去掉不影响核心的

**这个区分的价值：** 核心模型一旦确定就不再改动，所有新功能都是应用层的加法，架构永远稳定。

**本次案例：**
```
核心（不变）：原子、属性、关系 → 适用于一切语言
应用层（可扩展）：标签、轨迹、分析记录... → 随功能增长按需增加
```

---

## Phase 3: 验证与对抗

### ⑥ 用理论验证直觉

**原则：** 当你有一个直觉，不要只是"觉得对"，也不要因为找不到理论就放弃。去找理论验证。

**怎么做：**
- 把直觉表述为一个可验证的命题
- 寻找支持或反驳的研究/理论
- 如果找到支持，记录下来——直觉变成了"我知道为什么"
- 如果找到反驳，修正直觉

**本次案例：**
- "从句子出发比背单词好" → Krashen 的输入假说、Craik 的加工深度理论
- "人自己判断会不会很重要" → 元认知理论
- 两门语言的成绩作为实证验证（不是幸存者偏差，因为在不同语种上复现了）

---

### ⑦ 接受外部挑战，坚守关键判断

**原则：** 积极接受外部反馈，但要区分建议触及的是哪一层。

**怎么做：**
- 功能层面的建议（"先做 MVP"、"关系类型要收敛"）→ 评估后可采纳
- 哲学层面的挑战（"让 AI 自动入库"、"用户不需要自己判断"）→ 回到哲学锚点重新审视，不轻易让步
- 采纳的标准：这个建议是否违背了核心哲学？如果不违背，采纳；如果违背，拒绝并说明理由

**本次案例：**
- GPT 建议"AI 自动入库" → 拒绝（违背"元认知是内化的一部分"）
- GPT 建议"定位成 N2 语法工具" → 拒绝（格局太小，核心是通用语言知识系统）
- AI 建议"关系类型收敛为 5 种" → 采纳（执行层面的合理约束，不涉及哲学）
- AI 建议"MVP 先做最小闭环" → 采纳（优先级决策，不涉及哲学）

---

## Phase 4: 收敛

### ⑧ 追求优雅：最少概念，最广覆盖

**原则：** 好的设计不是能解决多少问题，而是用多少概念解决多少问题。

**判断标准：**
- 多一个概念是否冗余？→ 是则删
- 少一个概念是否不完整？→ 是则保留
- 这套概念能否覆盖相邻领域？→ 能则说明足够抽象

**本次案例：**
三个概念（原子、属性、关系）覆盖一切语言的知识表示。日语、英语、韩语、法语——同一个模型，只是属性类型和关系类型不同。

---

## 贯穿全程的方法：用对话打磨

不要试图一个人想清楚所有事情。用对话作为思考工具：

```
表达粗糙想法 → 接受挑战 → 发现盲区 → 修正 → 再表达 → 再挑战
```

每一次修正都是对话中自然发生的，不是预先计划的。关键是保持开放——允许自己的想法被改变，但只在理由充分时改变。

---

## 检查清单

在一个新项目开始时，按顺序回答以下问题：

- [ ] 我要解决的真实痛点是什么？（不是"我想做什么"）
- [ ] 这个产品存在的根本理由是什么？（一句话）
- [ ] 这个领域由哪些基本要素构成？
- [ ] 哪些是核心概念，哪些是应用层？
- [ ] 每个核心概念都通过了"删除测试"吗？
- [ ] 我的直觉有理论支撑吗？
- [ ] 核心概念能覆盖相邻场景吗？
- [ ] 多一个概念是否冗余？少一个是否不完整？
