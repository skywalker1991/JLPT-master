# JLPT Master — 技术架构文档

> 基于创世文档（VISION.md v0.4），定义系统的技术架构概念与选型原则。本文档不涉及具体实现细节（表字段、API 参数等），聚焦架构层面的决策和理由。

---

## 一、架构原则

### 1. 本地优先

系统必须能在用户本地通过 Docker 完整运行，不依赖任何自建云服务。

- 开源项目，用户应该能 `git clone` + `docker compose up` 即可启动
- 学习数据是私人的，本地存储更自然
- 减少运行成本

唯一允许的外部依赖是 **AI API**（如 Gemini、OpenAI），因为本地运行大模型对硬件要求过高。

### 2. 前后端职责分离

前端和后端是两个独立服务，**只通过 API 交互**，不共享数据库连接。

- **后端**：拥有所有数据和逻辑——数据库操作、AI 调用、词典查询、匹配、练习生成
- **前端**：只负责渲染和用户交互，通过调用后端 API 获取和提交数据

理由：
- 职责边界清晰，不存在"一次操作跨前后端来回跳"的问题
- 后端是完整的 API 服务，将来可以对接移动端、桌面端、或其他前端
- 开源协作友好，前端和后端贡献者可以完全独立工作

### 3. AI 是外部服务，不是系统核心

系统的核心是**知识库和学习流程**，AI 是可替换的外部能力提供者。

- AI 调用通过统一接口抽象，可切换不同供应商（Gemini / OpenAI / 本地模型）
- 即使 AI 服务不可用，知识库的浏览、复习、手动编辑仍然可用
- 不依赖特定 AI 供应商的私有特性

### 4. 数据是用户的

- 所有数据存在本地，用户完全拥有
- 支持数据导出（至少支持 JSON 格式的全量导出）
- 不做任何数据上传或遥测

### 5. 渐进式架构

MVP 的架构决策不能阻碍后续演进，但也不为未来过度设计。

- 数据库 schema 按创世文档的三个核心概念 + 应用层概念设计，不做"以后可能用到"的扩展
- 多语言支持通过数据隔离（不同语言的原子库独立），不在 MVP 实现

## 三、技术选型

### 前端

| 决策 | 选型 | 理由 |
|------|------|------|
| 构建工具 | **Vite** | 启动快、构建轻、无服务端渲染的多余负担 |
| UI 框架 | **React** | 组件生态丰富、动画库选择多 |
| 语言 | **TypeScript** | 类型安全，重构时有保障 |
| 样式 | **Tailwind CSS** | 实用优先，贡献者容易上手 |
| 路由 | **React Router** | SPA 客户端路由，轻量够用 |
| 动画 | **Framer Motion**（或同类） | 后期需要大量自定义交互动画，需要一个灵活的动画能力 |
| 状态管理 | **React 内置** | 不引入外部状态库，页面间状态通过 API 获取 |

为什么不用 Next.js：前后端严格分离后，前端是纯客户端 SPA，只调后端 API。Next.js 的核心能力（Server Actions、SSR/SSG、API Routes）全部用不上，反而增加了构建复杂度和 Server/Client 组件的心智负担。Vite + React 更轻、更直接、对自定义动画和交互效果的控制更灵活。

前端**不直接访问数据库**，所有数据通过后端 API 获取和提交。

### 后端

| 决策 | 选型 | 理由 |
|------|------|------|
| 框架 | **FastAPI (Python)** | 异步支持好、Pydantic 约束 AI 输出、Python AI/NLP 生态丰富 |
| 日语形态素分析 | **Janome（默认）/ MeCab（可选）** | Janome 纯 Python 零安装摩擦；MeCab 性能更强但需系统级安装 |
| AI 接口抽象 | **统一的 LLM Client 接口** | 底层可切换 Gemini / OpenAI / Ollama |

后端是系统的**唯一数据入口**，拥有所有业务逻辑和数据库操作。

### 关系型数据库

| 决策 | 选型 | 理由 |
|------|------|------|
| 数据库 | **PostgreSQL** | 成熟稳定、JSONB 支持灵活属性存储、Docker 一行启动 |

PostgreSQL 存储知识模型的三个核心概念（原子、属性、关系）和应用层数据（标签、轨迹、分析记录等）。

### 向量数据库

| 决策 | 选型 | 理由 |
|------|------|------|
| 向量数据库 | **Qdrant** | 开源、专业的向量检索引擎、Docker 一行启动、支持过滤和元数据 |

为什么用独立的向量数据库而不是 pgvector：
- Qdrant 是专业的向量检索引擎，检索质量和性能优于 pgvector
- 支持丰富的过滤条件（按类型、级别等过滤后再做语义检索）
- 关注点分离：结构化数据在 PostgreSQL，语义检索在 Qdrant
- Docker 部署，不增加用户的安装负担

Qdrant 主要用于：
- 语法原子的语义匹配（入库时去重和发现关联）
- 未来知识图谱中的"发现潜在关联"能力

词汇原子**不需要向量检索**——通过形态素分析还原辞书形后精确匹配即可。

### 日语词典

| 决策 | 选型 | 理由 |
|------|------|------|
| 词典数据 | **JMdict** | 最全面的开源日语词典，包含释义、读音、词性、JLPT 级别标注 |
| 角色 | **独立查询工具** | 用户主动查询时提供该词的完整信息（所有释义、所有读音），不参与分析流程 |

词典与 AI 的分工：

```
AI：告诉你这个词在这个句子里是什么意思（语境定位）
词典：告诉你这个词还有什么意思（全貌补充）
```

词典不参与分析、不自动填充属性、不做校验。它是用户可以随时调用的参考工具。

### AI 服务

| 决策 | 选型 | 理由 |
|------|------|------|
| 默认 AI | **Google Gemini** | 性价比高、结构化输出能力强、支持多模态（文本+图片） |
| 接口设计 | **可切换** | 用户可配置使用 OpenAI、Claude、或本地 Ollama |
| 向量生成 | **本地 Embedding 模型** | 开源模型（如 sentence-transformers）本地运行，不依赖外部 API，对硬件要求低 |

### TTS（第二阶段）

| 决策 | 选型 | 理由 |
|------|------|------|
| 默认 | **浏览器 Web Speech API** | 零成本、无需后端、日语支持够用 |
| 可选 | **Google Cloud TTS / OpenAI TTS** | 音质更好，用户自行配置 |

## 四、数据架构

### 概念到表的映射

分为知识模型和应用层两部分：

```
知识模型（核心三概念，不变）：
  原子（Atom）       →  PostgreSQL: atoms 表
  属性（Property）   →  PostgreSQL: atom_properties 表
  关系（Relation）   →  PostgreSQL: atom_relations 表

应用层（按需扩展）：
  标签（Tag）        →  PostgreSQL: atom_tags 表
  轨迹（Trace）      →  PostgreSQL: traces 表
  分析记录（Analysis）→  PostgreSQL: analyses 表
  其他应用概念        →  随功能增长按需增加

语义检索：
  语法原子向量       →  Qdrant: grammar_atoms collection
```

知识模型的三个概念是稳定的——原子（概念是什么）、属性（概念的特征）、关系（概念之间的联系）。这个结构适用于一切语言，不随功能扩展或语种增加而改变。应用层概念可以不断增加。

### 设计原则

**1. 原子表只存身份信息**

类型、key、基础释义、JLPT 级别。所有可变的、可追加的信息都通过属性表存储。

**2. 属性表是 append-only 的**

只做 INSERT，不做 UPDATE。每条属性带时间戳和来源。知识的生长轨迹可追溯。

**3. 关系是双向可查的**

A→B 存一条记录，查询时能从任意一端找到另一端。

**4. 轨迹表是行为日志**

每次用户与原子的交互记录一行。不做聚合存储，聚合在查询时计算。

**5. 向量只用于语法原子**

词汇原子用形态素分析还原辞书形后精确匹配。语法原子没有标准辞书形，通过 Qdrant 语义匹配。向量索引的规模可控。

**PostgreSQL 与 Qdrant 的一致性策略：** Qdrant 写入失败不阻塞入库。原子先写入 PostgreSQL（必须成功），向量写入 Qdrant 失败则后台重试。最坏情况是该原子暂时无法被语义匹配到，但精确匹配仍然可用，不影响核心功能。

## 五、AI 接口抽象

### 统一接口

后端定义一个 LLM Client 抽象，所有 AI 调用都通过它：

```
LLMClient（抽象接口）
├── analyze(content) → 结构化的词汇和语法列表
├── evaluate(source, user_input) → 评分和反馈
├── generate_quiz(atom_a, atom_b) → 对比选择题
└── extract_text(image) → OCR 文字

EmbeddingClient（本地，独立于 LLM）
└── embed(text) → 向量
```

具体实现：
```
├── GeminiClient      （默认）
├── OpenAIClient      （可选）
├── OllamaClient      （本地模型，可选）
└── 用户可自行扩展
```

### 配置方式

```
LLM_PROVIDER=gemini          # gemini | openai | ollama
LLM_MODEL=gemini-2.0-flash   # 具体模型名
LLM_API_KEY=xxx               # API key（ollama 不需要）
```

## 六、核心流程的技术路径（MVP 聚焦）

MVP 聚焦在**输入到结构化入库**这个阶段——这是整个系统的基础，也是用户最先感受到价值的环节。

### 核心设计原则：让 AI 在确定的范围内做事

AI 的输出质量取决于输入的约束程度。不要给 AI 一段原始文字让它自由发挥，而是先用本地工具构建确定性的结构，再让 AI 在这个结构内做判断。

```
本地预处理（确定性，快，免费）：
  分句、分词、词性标注、辞书形还原、变形类型识别
  → 约束了"有哪些词"

AI 在预处理结果的约束内做判断（理解性，消耗 API）：
  词汇筛选（从分词结果中过滤出值得学习的词）
  语法模式识别、语义分析、语感判断
  → 只做需要"理解"的事

词典（独立查询，用户主动调用）：
  用户对某个词想了解全貌时查询
  → 不参与分析流程
```

### 抽出准确性策略

**错误倾向：宁多勿少。** 用户是最终过滤器，多给选项让人选，比 AI 替人判断好。

| 策略 | 说明 |
|------|------|
| 尽可能多提取 | 仅过滤纯功能词（助词、助动词），其余都保留 |
| 不设数量上限 | 词汇和语法都不限制每句提取数量 |
| 人做最终筛选 | AI 提取候选，用户决定入库 |

### 入库准确性策略

**错误倾向：宁可多创建，不要误合并。**

词汇匹配（确定性高）：
```
形态素分析还原辞书形 → JMdict 确认 → 精确匹配 atoms 表
```

语法匹配（需要语义理解）：
```
精确匹配 atoms 表 → 未命中则 Qdrant 语义检索 → 分级处理：
  相似度 > 0.95   → 提示"已存在"，引导补充
  0.75 ~ 0.95     → 提示"发现相似"，建议建立关联
  < 0.75          → 作为新原子入库
```

**人是最终裁判。** 系统不自动合并，只呈现选项让用户确认。

### 完整流程

```
1. 用户输入
   ├── 粘贴文本
   └── 截图（OCR）
          │
          ▼
2. 即时返回 — 本地预处理（毫秒级，不消耗 AI）：
   ├── 截图 → AI OCR 提取文字（唯一需要 AI 的预处理步骤）
   ├── 分句
   └── 形态素分析 → 分词、词性标注、辞书形还原
   → 前端立刻展示分词结果，用户不等待
          │
          ▼
3. 流式返回 — AI 分析（秒级，逐条补充）：
   输入：原文 + 分词结果
   输出：词汇筛选 + 语法识别 + 翻译 + 语感/用法等理解性信息
   → 逐条流式补充到已有的分词结果上，标记为 AI 生成
          │
          ▼
4. 追问交互 — 用户驱动的知识生长（Schema 约束）：
   系统提供结构化追问模板：[对比] [用法] [变形] [例句] [自由追问]
   每种模板对应专用 prompt + 固定输出 Schema
   追问携带完整上下文（原始分析 + 所有追问历史）
   整条探索路径存入分析记录
   → 追问的输出同样是结构化的，可以可靠提取原子和关系
          │
          ▼
5. 用户逐条判断（元认知过程）：
   不标记匹配状态——让用户自己判断"我会不会"
   ├── 这个我不会 → 点击 [入库]
   └── 这个我会 → 跳过
          │
          ▼
6. 入库时匹配检查（用户点击 [入库] 后才触发）：
   ├── 词汇 → 辞书形精确匹配 atoms 表
   └── 语法 → 精确匹配 → 未命中则 Qdrant 语义检索
          │
          ├── 不存在 → 创建原子 + 词典属性 + AI 属性 + trace(added)
          ├── 已存在 → 提示"已在库中"，引导补充新属性 + trace(duplicate_attempt)
          └── 发现相似 → 展示相似原子 + AI 对比说明
                        用户确认：[是同一个] → 合并补充
                                  [不同，建立关联] → 新原子 + 关系
                                  [不同，无关] → 新原子
```

### Schema 驱动的可预测性

整个流程的可靠性依赖于**每一步输出都有确定的结构**。从 AI 输出到知识库写入，每一步都通过 Schema 约束：

```
AI 输出 Schema（Pydantic 约束）
  → 每种操作（分析、对比追问、用法追问、题型模板）各有专用 Schema
      ↓
校验层
  → 词典校验 + 规则校验
      ↓
转换层
  → AI 输出格式 → 知识库写入格式
      ↓
知识库写入 Schema
  → 原子、属性、关系各有固定的写入结构
```

只有每步输出可预测，最终才能可靠地组装成知识网络，并有利于显示和查询。Schema 的具体定义在详细设计文档中。

### 输出校验层

AI 的输出在写入知识库之前必须经过校验，不能 AI 说什么就是什么。

```
AI 输出 → 校验层 → 通过后才呈现给用户 → 用户确认后才写入
```

校验策略按成本从低到高：

| 层级 | 方式 | 说明 |
|------|------|------|
| 词典校验 | 辞书形是否存在、读音是否正确、JLPT 级别是否准确 |
| 规则校验 | 输出格式是否完整、必填字段是否为空 |
| 交叉验证（预留） | 多模型验证关键判断（语感、对比关系等），MVP 不实现 |
| 人工确认 | 用户审核后才入库，始终是最终裁判 |

校验层是架构中的独立环节，未来可以不断增加新的校验规则，而不影响前后的流程。

### AI 降级策略

AI 失败时，按场景区分处理：

- **自由文本分析**：AI 失败 → 返回本地预处理结果（分词、词性、辞书形、JMdict 释义和级别），用户仍然可以浏览和入库
- **JLPT 题型分析**：完全依赖 AI（选项对比、排序方案、阅读分析），AI 失败 → 提示失败，建议重试，无法降级
- **追问**：AI 失败 → 提示失败，已有分析结果不受影响
- **入库匹配**：Qdrant 失败 → 降级为精确匹配，语义检索暂不可用

### 各层职责

```
前端：输入界面 → 展示结构化结果 → 用户选择交互 → 展示入库结果
后端：预处理（分词+词典）→ AI 分析 → 匹配检查 → 入库写入
PostgreSQL：原子、属性、关系的存储
Qdrant：语法原子的语义匹配检索
AI：OCR、词汇筛选、语法识别、语感判断、对比说明（在预处理约束下工作）
JMdict：独立查询工具（用户主动调用，不参与分析流程）
形态素分析：分词、词性标注、辞书形还原（AI 的输入约束）
```

## 七、部署架构

### Docker Compose

所有服务通过 Docker Compose 一键启动：

```
services:
  postgres:   PostgreSQL（结构化数据）
  qdrant:     Qdrant（向量检索）
  backend:    FastAPI（应用逻辑 + API）
  frontend:   Vite + React（用户界面）
```

启动流程：

```bash
# 1. 配置 AI API key
cp .env.example .env  # 编辑填入 API key

# 2. 启动所有服务
docker compose up -d
```

### 开发模式

开发时可以只用 Docker 跑数据库，前后端本地运行：

```bash
# 启动数据库
docker compose up -d postgres qdrant

# 后端
cd backend && python -m uvicorn main:app --reload

# 前端
cd frontend && pnpm dev
```

### 开源友好

- 前后端通过 API 分离，贡献者可以只关注自己擅长的部分
- AI Client 接口抽象，社区可以贡献新的供应商适配
- 数据库 schema 迁移脚本版本化管理
- JMdict 词典数据随项目分发或首次启动时自动下载

### 架构演进路径

**MVP → 第二阶段**需要新增的技术能力：
- 输出练习（翻译造句 + AI 评分、对比选择题）
- 基于轨迹的复习优先级
- TTS 集成（Web Speech API，前端即可完成）
- 字幕文件解析（SRT/ASS 解析库，后端新增）
- YouTube 字幕提取（浏览器插件或 API）

**第二阶段 → 第三阶段**需要新增的技术能力：
- 知识图谱渲染（D3.js / Cytoscape.js，前端新增）
- 图谱驱动练习出题（后端扩展）

每个阶段的新增能力都是**加法**，不需要改动已有架构。
